# Android Validation Checklist

This checklist is for the current native Android mainline in `app/`.

Goal:

- verify the local notification + notebook learning loop on a real device
- confirm the current beta build is usable even if emulator-only warnings still exist

## Scope

This checklist is intentionally narrow.

It covers only the most important user-facing loop:

1. enable push
2. receive a word notification
3. mark a word as known or unknown
4. verify notebook behavior
5. verify unlock refresh behavior
6. verify push-off behavior

## Preconditions

- Install the latest debug or beta build.
- Grant notification permission on first launch.
- Open the app once and confirm the home screen is visible.
- Keep at least one local word library enabled.

## Pass Criteria Summary

The build can be treated as functionally acceptable if all of the following are true:

- a word notification appears after push is enabled
- tapping `不认识` causes the word to appear in the notebook
- tapping `认识` does not cause repeated notification spam
- unlocking the device refreshes at most once per expected cycle
- disabling push stops new notifications

## Test 1: Home Screen State

Steps:

1. Open the app.
2. Stay on the home page.
3. Observe push status, selected library, and notebook count.

Expected:

- the home page loads without crashing
- the push toggle is visible and responsive
- the selected library text is readable
- the notebook count is shown

Record:

- `Pass` or `Fail`
- note the displayed notebook count

## Test 2: Enable Push

Steps:

1. Turn on the push toggle.
2. Wait a few seconds.

Expected:

- a success message appears
- a word notification is posted
- the current word is readable in the notification
- the notification contains `认识` and `不认识`

Record:

- whether notification appeared
- how long it took to appear

## Test 3: Unknown -> Notebook

Steps:

1. In the notification, tap `不认识`.
2. Return to the app.
3. Switch to the notebook page.

Expected:

- the current notification is handled cleanly
- the notebook page opens normally
- the just-marked word is visible in notebook data
- notebook count increases if the word was not already present

Record:

- the word you tapped
- whether it appeared in notebook
- whether count changed

## Test 4: Notebook Display

Steps:

1. Stay on the notebook page.
2. Confirm the visible word card and the side list.
3. Tap one or two words in the list if available.

Expected:

- the notebook title is shown
- the current word card shows word, phonetic, translation, and example
- tapping a notebook item updates the displayed detail
- pronunciation button does not crash the app

Record:

- whether the page displayed data correctly
- any missing fields such as library or part of speech

## Test 5: Known -> Unlock Refresh

Steps:

1. Go back to a live word notification.
2. Tap `认识`.
3. Lock the device.
4. Unlock the device once.

Expected:

- the handled notification disappears or is replaced cleanly
- after unlock, the next word appears once
- the app does not spam multiple rapid updates

Record:

- whether exactly one refresh was observed
- whether the same word repeated unexpectedly

## Test 6: Screen On vs Unlock Timing

Steps:

1. With push enabled, turn the screen off.
2. Wake the screen without fully unlocking.
3. Observe whether a new word appears.
4. Then unlock.

Expected:

- waking the screen alone should not cause noisy repeated refreshes
- unlocking may trigger a single refresh according to current logic

Record:

- what happened on wake only
- what happened on full unlock

## Test 7: Disable Push

Steps:

1. Return to the home page.
2. Turn off the push toggle.
3. Wait and lock/unlock once.

Expected:

- a success message appears
- new word notifications stop
- no new notification should be generated by normal unlock after shutdown

Record:

- whether notifications fully stopped

## Test 8: Reboot Recovery

Steps:

1. Enable push.
2. Reboot the device.
3. Unlock and wait briefly.
4. Then disable push and reboot again if practical.

Expected:

- when push is enabled, scheduling should recover after boot
- when push is disabled, boot should not silently reactivate push behavior

Record:

- result after reboot with push enabled
- result after reboot with push disabled

## Known Current Risk

At the time of writing, the emulator may still show a foreground-service-type warning in system logs.

Interpretation:

- if user-visible behavior is normal on a real device, this warning should be treated as a compatibility tail rather than immediate product failure
- if the same warning appears on a real device and causes notification instability, `WordService` startup remains the next engineering priority

## Final Decision Template

Use this summary after testing:

- Home screen: `Pass` / `Fail`
- Enable push: `Pass` / `Fail`
- Unknown -> notebook: `Pass` / `Fail`
- Notebook display: `Pass` / `Fail`
- Known -> unlock refresh: `Pass` / `Fail`
- Screen wake / unlock timing: `Pass` / `Fail`
- Disable push: `Pass` / `Fail`
- Reboot recovery: `Pass` / `Fail`

Overall release judgment:

- `Accept for beta`
- `Accept with known warning`
- `Do not accept yet`
